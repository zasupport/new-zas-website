## 1. MANDATORY PRE-FLIGHT — BEFORE EVERY BLOG

### 1.0 Audit before building (no duplicate work)

Before creating, modifying, or deploying any blog post, component, schema, or page:

1. **Check what already exists.** Search the Next.js repo, Sanity CMS, live site (zasupport.com), and any previously delivered packages. Map what's done vs what's outstanding before starting.
2. **Read existing files in full** before replacing them. Never overwrite without checking. CLAUDE.md files are always appended, never overwritten.
3. **Compare existing vs proposed.** Identify additions, corrections, regressions, and stylistic changes explicitly. Preserve anything not contradicted by the new spec — especially custom logic, edge case handling, and workshop-specific data.
4. **Only update if newer AND proven.** A new version replaces the existing version only if it reflects more recent information AND contains patterns verified to work (tested, documented in trusted sources, or running in production). Mark unverified-but-newer changes as `// EXPERIMENTAL` and ask before deploying.
5. **Ask when unsure, never guess.** If ambiguity arises, ask Courtney directly with numbered options and a clear default. While waiting for an answer, continue every other task that doesn't depend on the answer. Never sit idle.

### 1.1 Per-post pre-flight checks

1. **Cannibalisation check** — Search existing posts at zasupport.com/blog. If >60% topic overlap, UPDATE that post instead.
2. **Keyword validation** — Confirm primary keyword has search volume (check People Also Ask + autocomplete).
3. **SERP analysis** — Check top 3 ranking pages for word count, headings, content gaps.
4. **Author assignment** — Every post MUST have named author: courtney-bentley or mary-blount. Never anonymous.
5. **Duplicate title check** — No two posts may target the same primary keyword.

## 9. GEO OPTIMISATION (AI Search Citation)

Google AI Overviews, ChatGPT, and Perplexity use RAG to cite sources. Optimise for citation:

1. **Key Takeaways block** — 3-5 declarative bullet points in first 200 words
2. **Chunk-friendly structure** — Clear H2/H3 hierarchy, no buried information in long paragraphs
3. **Direct answers** — 40-60 word answer immediately below each question heading
4. **Entity schema** — Person + Organization + sameAs + knowsAbout makes source identity unambiguous
5. **Original data** — Repair counts, pricing, success rates that only ZA Support can provide
